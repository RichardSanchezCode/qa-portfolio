# QA Analysis Notes

This document contains QA observations, behavioral analysis, and lessons learned while testing REST APIs using Postman.

---

## Scenario 1: Create User Without Required Name

### Objective

Analyze the API behavior when a required field is omitted.

### Request

```json
{
    "username": "RichardQA",
    "email": "richard@example.com"
}
```

### Observed Behavior

- Status Code: **201 Created**
- The API generated a new user even though the `name` field was missing.

### Expected Behavior (Production System)

If the `name` field is mandatory according to the requirements, the API should return:

- **400 Bad Request**
- An error message indicating that the `name` field is required.

### QA Analysis

JSONPlaceholder is a training API and does not validate business rules. Therefore, this behavior is expected for learning purposes.

In a production environment, this behavior should be reported as a defect if the requirements specify that `name` is a required field.

### Lesson Learned

A QA engineer must always compare the system behavior against the requirements, not only against the HTTP response.

## Scenario 2: DELETE Request Returns an Empty Response

### Objective

Analyze the API behavior after sending a DELETE request.

### Request

```http
DELETE /users/1
```

### Observed Behavior

- Status Code: **200 OK**
- Response Body:

```json
{}
```

### Expected Behavior (Production System)

A production API may return:

- **200 OK** with a confirmation message.
- **204 No Content** with an empty response body.

After the deletion, a new request to:

```http
GET /users/1
```

would typically return:

- **404 Not Found**

because the resource no longer exists.

### QA Analysis

JSONPlaceholder simulates the DELETE operation but does not permanently remove data. Therefore, a subsequent GET request still returns the original resource.

A QA engineer must understand the limitations of mock APIs and distinguish between simulated behavior and production behavior.

### Lesson Learned

Always validate API behavior against the requirements and understand whether the testing environment is a mock service or a production-like environment.

## Scenario 3: Query Parameter Returns No Results

### Objective

Analyze the API behavior when a query parameter does not match any existing resource.

### Request

```http
GET /users?id=999
```

### Observed Behavior

- Status Code: **200 OK**
- Response Body:

```json
[]
```

### Expected Behavior (Production System)

The API should return **200 OK** with an empty array because the request was valid, but no matching records were found.

A **404 Not Found** is generally not expected for filtered collection endpoints.

### QA Analysis

The request was processed successfully. The empty array indicates that no users matched the filter.

This behavior is different from requesting a specific resource such as:

```http
GET /users/999
```

which would typically return **404 Not Found** if the resource does not exist.

### Lesson Learned

A successful search with no matching records usually returns an empty collection (`[]`) instead of an error.

## Scenario 4: Understanding HTTP Headers

### Objective

Understand the purpose of HTTP Headers and their impact on API requests.

### Headers Reviewed

- Content-Type
- Accept
- User-Agent
- Host
- Content-Length
- Postman-Token

### Custom Header Tested

```text
X-Environment: QA
```

### Observed Behavior

The custom header was successfully sent with the request.

JSONPlaceholder ignored the header and processed the request normally, returning **201 Created**.

### QA Analysis

Not every API uses custom headers. Mock APIs such as JSONPlaceholder ignore headers that are not part of their implementation.

In production environments, headers are often required for authentication, environment selection, content negotiation, versioning, and security.

### Lesson Learned

A QA engineer must understand which headers are automatically generated by Postman and which headers must be manually configured according to the API specification.

## Scenario 5: Dynamic Variables

### Objective

Understand how to reuse dynamic data between API requests.

### Description

The response generated by the POST request was used to automatically store the generated user ID as a Collection Variable.

```javascript
pm.collectionVariables.set("userId", jsonData.id);
```

### Observed Behavior

The generated ID was successfully stored and reused by subsequent requests.

### QA Analysis

Dynamic variables eliminate hardcoded values and make API tests reusable, maintainable, and easier to scale.

### Lesson Learned

A QA engineer should avoid hardcoded values whenever possible and reuse dynamic data generated during test execution.

## Scenario 6: Environment Variables

### Objective

Understand how Environment Variables simplify API execution across multiple environments.

### Description

A QA Environment was created using the variable:

```text
baseUrl
```

All requests were updated to use:

```text
{{baseUrl}}
```

instead of hardcoded URLs.

### Observed Behavior

Changing the selected Environment automatically updates the API endpoint without modifying the requests.

### QA Analysis

Environment Variables reduce maintenance effort and allow the same collection to be executed against Development, QA, UAT, or Production environments.

### Lesson Learned

Using Environment Variables is considered a best practice for scalable and maintainable API testing.

## Scenario 7: Data Driven Testing

### Objective

Understand the concept of Data Driven Testing using CSV files.

### Description

The POST request body was parameterized using variables.

```json
{
    "name": "{{name}}",
    "username": "{{username}}",
    "email": "{{email}}"
}
```

A CSV file containing multiple user records was prepared for execution through the Collection Runner.

Due to the limitations of the free version of Postman, the CSV execution could not be completed.

### QA Analysis

Data Driven Testing separates test logic from test data, allowing the same test case to be executed multiple times using different datasets.

This approach improves scalability and reduces duplicated test cases.

### Lesson Learned

Parameterized tests increase test coverage, improve maintainability, and are widely used in professional API automation projects.
